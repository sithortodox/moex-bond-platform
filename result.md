# Диагностический отчёт: проблемы с SSH-доступом к VPS 45.67.230.123

**Дата:** 2026-05-07  
**Клиентский IP:** 138.124.65.127  
**Сервер:** 45.67.230.123 (Ubuntu, OpenSSH 9.6p1)  
**Порт:** 22  

---

## Краткое описание проблемы

SSH-подключение к серверу 45.67.230.123:22 невозможно. TCP-соединение устанавливается, SSH-баннер сервера получается, но handshake (key exchange) никогда не завершается. Клиентские пакеты после TCP-connect не достигают сервера.

## Диагноз

Проблема вызвана **нарушением двунаправленного прохождения SSH-пакетов** на сетевом уровне:

1. **От сервера к клиенту** — пакеты проходят (баннер, kex_init получены корректно за 0.1с).
2. **От клиента к серверу** — пакеты после TCP-connect **не достигают сервера** (соединение остаётся живым, сервер ждёт, но ответа нет).
3. Только порт 22 открыт для TCP-connect. Все остальные порты заблокированы.

### Возможные причины (по вероятности)

| # | Причина | Вероятность | Обоснование |
|---|---------|-------------|-------------|
| 1 | **Firewall/iptables на сервере** блокирует входящие SSH-пакеты после SYN/ACK | Высокая | iptables может разрешать только TCP SYN (new connections), но блокировать ESTABLISHED пакеты с payload |
| 2 | **ISPmanager firewall** с некорректными правилами | Высокая | ISPmanager управляет файрволом; неверная конфигурация может фильтровать SSH payload |
| 3 | **Upstream firewall (провайдер хостинга)** с DPI | Средняя | Провайдер может фильтровать SSH-трафик на уровне DPI, пропуская только initial packets |
| 4 | **conntrack/nat** не корректно отслеживает SSH-соединения | Низкая | Соединение устанавливается, но stateful firewall теряет состояние |

## Рекомендации для провайдера / администратора

### Проверить на сервере (через ISPmanager или консоль):

```bash
# 1. Проверить iptables правила
iptables -L -n -v
iptables -t nat -L -n -v
iptables -t mangle -L -n -v

# 2. Проверить nftables
nft list ruleset

# 3. Проверить ufw
ufw status verbose

# 4. Проверить логи блокировок
dmesg | grep -i "drop\|reject\|block" | tail -20
journalctl -k | grep -i "drop\|reject" | tail -20

# 5. Проверить SSH конфигурацию
cat /etc/ssh/sshd_config | grep -v "^#" | grep -v "^$"

# 6. Проверить fail2ban
fail2ban-client status
fail2ban-client status sshd

# 7. Временно открыть все порты для диагностики
iptables -I INPUT -s 138.124.65.127 -j ACCEPT
# или
ufw allow from 138.124.65.127

# 8. Проверить conntrack
conntrack -L | grep 138.124.65.127
cat /proc/sys/net/netfilter/nf_conntrack_max
cat /proc/sys/net/netfilter/nf_conntrack_count
```

### Проверить ISPmanager:

1. Зайти в **https://45.67.230.123:1500**
2. Раздел **Безопасность → Файрвол**
3. Проверить правила на предмет блокировки входящих SSH-пакетов
4. Убедиться, что правило для порта 22 разрешает **ESTABLISHED,RELATED** трафик

### Минимальное правило iptables для корректной работы SSH:

```bash
iptables -I INPUT 1 -p tcp --dport 22 -m state --state NEW,ESTABLISHED,RELATED -j ACCEPT
iptables -I INPUT 1 -m state --state ESTABLISHED,RELATED -j ACCEPT
```

### Открыть порты для веб-доступа:

```bash
iptables -I INPUT -p tcp --dport 80 -j ACCEPT
iptables -I INPUT -p tcp --dport 443 -j ACCEPT
iptables -I INPUT -p tcp --dport 8501 -j ACCEPT
iptables -I INPUT -p tcp --dport 1500 -j ACCEPT
```

## Сводка

| Параметр | Значение |
|----------|----------|
| TCP connect :22 | Работает |
| SSH баннер от сервера | Получен (0.1с) |
| Kex Init от сервера | Получен (0.1с) |
| SSH handshake завершается | **НЕТ** |
| Клиентские пакеты до сервера | **Не доходят** |
| Соединение разрывается | **НЕТ** (живёт 30+ сек) |
| Другие порты | Все заблокированы |
| Пакетная потеря (mtr) | 20% на последнем хопе |

**Основной вывод:** На сервере или на стороне хостинг-провайдера действует firewall, который:
1. Разрешает TCP SYN на порт 22 (connect проходит)
2. Разрешает исходящие от сервера пакеты (баннер получен)
3. **Блокирует входящие к серверу SSH payload-пакеты** (handshake не завершается)
4. Блокирует все остальные TCP-порты

Требуется проверка и корректировка правил файрвола на сервере (iptables/nftables/uffw/ISPmanager) и/или на стороне провайдера хостинга.
