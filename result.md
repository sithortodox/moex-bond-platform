# Диагностический отчёт: SSH-доступ к VPS 45.67.230.123

**Дата:** 2026-05-07

## Проблема

SSH-подключение к 45.67.230.123:22 невозможно ни напрямую, ни через SOCKS5 прокси (Hiddify, 127.0.0.1:12334).

## Ключевые факты

| Тест | Результат |
|------|-----------|
| TCP connect :22 (напрямую) | OK |
| TCP connect :22 (через SOCKS5) | OK |
| SSH баннер от сервера (напрямую) | Получен (0.1с) |
| SSH баннер от сервера (через SOCKS5) | Получен |
| Kex Init от сервера (напрямую) | Получен (0.1с, 1120 байт) |
| Kex Init от сервера (через SOCKS5) | Получен |
| Клиентский Kex Init → ответ сервера | **НЕТ** (таймаут 25с) |
| Мусорные пакеты → ответ сервера | **Да** (SSH_MSG_DISCONNECT) |
| Все порты кроме 22 | Заблокированы |

## Диагноз: Deep Packet Inspection (DPI)

1. TCP-соединение устанавливается (напрямую и через прокси)
2. Сервер отправляет баннер и kex_init — доходит мгновенно
3. Клиент отправляет kex_init — **сервер его никогда не получает** (таймаут, но соединение живёт 30+ сек)
4. Отправка мусорных (не-SSH) пакетов — **сервер получает и отвечает** (SSH_MSG_DISCONNECT: PROTOCOL_ERROR)
5. Прокси не помогает — поведение идентично

**Вывод:** На стороне хостинг-провайдера работает DPI-фильтр, который:
- Пропускает TCP SYN/ACK (соединение устанавливается)
- Пропускает исходящие от сервера пакеты (баннер, kex_init получены)
- **Блокирует входящие к серверу корректные SSH-пакеты** (kex_init не доходит)
- Пропускает некорректные/мусорные данные (они доходят, сервер отвечает DISCONNECT)

Это подтверждается тем, что подключение с российских IP работает (по словам владельца), а с иностранных — нет. Провайдер фильтрует SSH на уровне содержимого пакетов.

## Решение

### Вариант 1: Сменить порт SSH (рекомендуется)

Через ISPmanager (https://45.67.230.123:1500):

1. Открыть **Настройки → Службы → SSH**
2. Изменить порт с 22 на, например, 22222
3. Или через консоль:

```bash
# В ISPmanager терминале:
sed -i 's/^#Port 22/Port 22222/' /etc/ssh/sshd_config
sed -i 's/^Port 22/Port 22222/' /etc/ssh/sshd_config
systemctl restart sshd
```

После этого подключаться: `ssh -p 22222 root@45.67.230.123`

### Вариант 2: Обратиться к хостинг-провайдеру

Сообщить провайдеру о DPI-фильтрации SSH на порту 22 и попросить:
- Отключить DPI для SSH-трафика
- Или разрешить SSH на альтернативном порту

### Вариант 3: Использовать ISPmanager терминал

Деплой можно выполнить через встроенный терминал ISPmanager (https://45.67.230.123:1500), который работает через HTTPS, а не через SSH.
